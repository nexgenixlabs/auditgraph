### 2.4 FINDING: Group/Membership Accumulation Bug

**entra_groups per run:**

| discovery_run_id | status    | group_count |
|------------------|-----------|-------------|
| 158              | completed | 2452        |
| 154              | completed | 2443        |
| 151              | archived  | 2443        |
| 149              | archived  | 2443        |
| 137              | archived  | 2443        |
| 134              | archived  | 2442        |
| 129              | archived  | 2442        |
| 128              | archived  | 2442        |
| 126              | archived  | 2442        |
| 123              | archived  | 2442        |
| 122              | archived  | 2442        |
| **TOTAL**        |           | **26,876**  |

**entra_group_memberships per run:**

| discovery_run_id | status    | membership_count |
|------------------|-----------|------------------|
| 158              | completed | 1897             |
| 154              | completed | 1886             |
| 151              | archived  | 1885             |
| 149              | archived  | 1886             |
| 137-122          | archived  | ~1885 each       |
| **TOTAL**        |           | **20,751**       |

**Bug:** Archived runs correctly delete `identities` rows (all show 0) but
do NOT clean `entra_groups` or `entra_group_memberships`. This causes
unbounded accumulation:

- Expected groups (2 completed runs): ~4,895
- Actual groups: 26,876 (**5.5x inflation**)
- Expected memberships (2 completed runs): ~3,783
- Actual memberships: 20,751 (**5.5x inflation**)

**Root cause:** The run archival logic deletes identity rows for archived
runs but does not clean entra_groups or entra_group_memberships. Both tables
have `ON DELETE CASCADE` on `discovery_run_id` FK to `discovery_runs`, but
the archival process sets `status='archived'` without deleting the run
row — so the cascade never fires.

**Impact:** Storage bloat proportional to retained archived runs. Each
un-cleaned run adds ~2,450 group rows + ~1,886 membership rows. Over time
this is O(n) growth with no upper bound.

## Findings

### Finding 1: Group/Membership Accumulation (NEW)

**Severity:** Medium
**Ticket:** File as AG-74 (or reopen if AG-74 already exists)
**Summary:** `entra_groups` and `entra_group_memberships` rows are not
cleaned when discovery runs are archived, while `identities` rows ARE cleaned.
5.5x storage inflation observed on org=10 (26,876 group rows vs ~4,895
expected).
**Fix:** Add `DELETE FROM entra_groups WHERE discovery_run_id = <archived_id>`
and `DELETE FROM entra_group_memberships WHERE discovery_run_id = <archived_id>`
to the run archival logic, matching the existing identity cleanup pattern.

### Finding 2: Identity Count Delta vs Closing Comment

**Severity:** Informational (not a bug)
**Summary:** AG-72 closing comment reported 2570 human users; current latest
run shows 966. Delta is expected — Entra directory membership changes over
time. The fix (plan-tier cap removal) is confirmed working: humans are
persisted without truncation.
