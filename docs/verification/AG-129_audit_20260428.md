# AG-129: Pre-Migration Audit — `identities.organization_id`

**Date**: 2026-04-28
**Ticket**: AG-129 (Deliverable 1 of 6)
**CWE**: CWE-639 (IDOR), CWE-284, CWE-200, CWE-863
**OWASP**: A01:2021, API1:2023

---

## Executive Summary

The `identities` table **already has** an `organization_id` column (integer, nullable), fully populated (8129/8129 rows, 0 NULLs), with strict RLS policies active. However, the column lacks:
- NOT NULL constraint
- Index (zero indexes on `organization_id`)
- FK constraint to `organizations(id)`
- Consistency trigger (org_id must match `discovery_runs.organization_id`)

The codebase uses **transitive scoping** (identities scoped via `discovery_run_id → discovery_runs.organization_id`) in ~95% of queries. The Phase B code switchover will add direct `organization_id` filtering and eliminate the transitive join overhead.

### Scoping Pattern (Current)

All properly-scoped queries follow this chain:
```
org_id = _org_id()  (from JWT)
  → run_ids = _latest_run_ids(cursor, org_id, connection_id)
    → WHERE i.discovery_run_id = ANY(%s)  (parameterized)
```

This works but is:
1. **Indirect** — 2-hop join through `discovery_runs` for every query
2. **Performance tax** — subquery overhead on every identity read
3. **Fragile** — if `_latest_run_ids()` returns empty (no completed runs), fallback paths may leak

**Key finding**: There is exactly **ONE** INSERT path into `identities` (`save_identity()`), and it already writes `organization_id` from `self._organization_id`. The SCIM/SAML JIT paths write to the `users` table, NOT `identities`.

**ON CONFLICT behavior**: The UPSERT does NOT update `organization_id` on conflict (`organization_id` is only in the INSERT, not in the `DO UPDATE SET` clause). This is correct — org_id is immutable after creation.

## 6. Migration Plan (Recommended)

### Phase A: Schema (Additive, Non-Breaking)

```sql
-- 1. NOT NULL constraint (safe — 0 NULLs exist)
ALTER TABLE identities ALTER COLUMN organization_id SET NOT NULL;

-- 2. Index (most impactful single change)
CREATE INDEX CONCURRENTLY idx_identities_org_id ON identities(organization_id);

-- 3. Composite index for hot queries
CREATE INDEX CONCURRENTLY idx_identities_org_risk
  ON identities(organization_id, is_microsoft_system, risk_level)
  WHERE is_microsoft_system = false;

-- 4. FK constraint (with validation)
ALTER TABLE identities
  ADD CONSTRAINT fk_identities_organization
  FOREIGN KEY (organization_id) REFERENCES organizations(id);

-- 5. Consistency trigger
CREATE OR REPLACE FUNCTION trg_identities_org_consistency()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.organization_id != (
    SELECT organization_id FROM discovery_runs WHERE id = NEW.discovery_run_id
  ) THEN
    RAISE EXCEPTION 'identities.organization_id (%) does not match discovery_runs.organization_id',
      NEW.organization_id;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_identities_org_check
  BEFORE INSERT OR UPDATE ON identities
  FOR EACH ROW EXECUTE FUNCTION trg_identities_org_consistency();
```

### Phase B: Code Switchover

1. Fix 10 NEEDS_SCOPE queries (add `AND organization_id = %s`)
2. Add `organization_id` filter alongside `discovery_run_id = ANY(%s)` in hot paths (belt-and-suspenders)
3. Copilot service: pass `org_id` from authenticated handler context
4. `@requires_org_id` decorator on all DB methods that query identities

### Phase C: Registry + Cleanup

1. Update MEMORY.md to note identities now has direct org_id scoping
2. Remove "identities has NO tenant_id" note from memory
3. Document the dual-scoping pattern (org_id + run_ids) as the standard
