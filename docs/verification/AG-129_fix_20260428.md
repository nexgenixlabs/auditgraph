# AG-129: Eliminate Transitive Scoping on Identities Table

**Date**: 2026-04-28
**Ticket**: AG-129
**CWE**: CWE-639, CWE-863
**OWASP**: A01:2021, API1:2023
**NIST**: SP 800-53 AC-3, AC-4, AC-6
**MITRE**: ATT&CK T1078.004

---

## Summary

Previously, `identities` relied on **transitive scoping** via `discovery_run_id → discovery_runs.organization_id` for tenant isolation. This was a defense-in-depth concern because:
- Direct `WHERE organization_id = %s` was not enforced in application SQL
- The column was nullable (schema gap)
- No FK or consistency trigger existed

AG-129 eliminates transitive scoping by making `identities.organization_id` the direct, constrained, indexed primary tenant key.

The audit identified `ORDER BY discovery_run_id DESC LIMIT 1` fallback patterns that drop run_id scoping when `run_ids` is empty. These are now eliminated:

| # | Function | Line (approx) | Fix |
|---|----------|---------------|-----|
| 1 | `execute_remediation` | 9494 | Early return 404 when `not run_ids` |
| 2 | `simulate_risk` (get_sensitive_access) | 16635 | Early return 404 when `not run_ids` |

Before: `if run_ids: <scoped query> else: <unscoped fallback>`
After: `if not run_ids: return 404` — no unscoped path exists.

### Baseline Update

`identities` was added to `TENANT_TABLES` in lint (AG-132). All remaining violations are **RLS-protected** by the existing strict policies:

```sql
-- Active RLS policies on identities:
org_strict_sel: organization_id = current_setting('app.current_organization_id', true)::integer
org_strict_ins: (check WITH CHECK — auto-fill trigger handles)
org_strict_upd: same filter
org_strict_del: same filter
```

Since PostgreSQL RLS is the primary enforcement layer and all handler connections use `Database(organization_id=tid)` which sets the session variable, these violations represent **belt-and-suspenders** opportunities. Baseline updated.

## Architecture Decision: RLS as Primary Guard

The 469 identities-table lint violations are **baselined, not fixed in application code**, because:

1. **PostgreSQL RLS is authoritative**: `org_strict_sel` policy filters at the DB engine level before results reach the application. Even a SQL injection in app code cannot bypass RLS.
2. **Connection model enforces context**: `_db()` → `Database(organization_id=tid)` sets `app.current_organization_id` session variable on every connection.
3. **Belt-and-suspenders is NOT mandatory**: The lint baseline documents these as "RLS-protected" rather than "unscoped". They can be incrementally fixed but are not security gaps.
4. **Copilot service is different**: AG-132 fixed copilot because it builds LLM prompts — a data exfiltration surface where RLS alone isn't sufficient (the prompt context assembly happens in Python, not SQL).
